# Animal Simulation Implementation Reference

이 문서는 현재 프로젝트에서 새 동물을 추가하거나 기존 동물 행동을 수정할 때 사용하는 공통 구현 기준이다.

이 파일의 목적은 사람이 읽는 일반 설명문이 아니라, 새 채팅에서 외부 논문 요약이나 행동 정리 텍스트가 들어왔을 때 VS Code Copilot이 바로 현재 프로젝트 구조에 맞는 구현으로 번역할 수 있게 하는 것이다.

이 문서에는 종별 규칙, 종별 수치, 종별 예외 처리를 넣지 않는다. 특정 동물에만 적용되는 내용은 이 파일이 아니라 해당 동물 코드나 별도 작업 메모로 관리한다.

## 목적

- 외부 행동 설명을 구현 가능한 개체 규칙으로 변환한다.
- 새 동물을 추가할 때 공통 구현 기준으로 사용한다.
- 기존 동물을 수정할 때도 종별 임시 규칙이 공용 계층으로 새어 나오지 않게 한다.

## 이 문서가 다루는 범위

- 다루는 것:
  - 개체 기반 행동 해석 원칙
  - 수치와 상태 설계 원칙
  - 스프라이트와 표현 계층의 공통 규칙
  - 현재 프로젝트에서 재사용해야 하는 공용 구현 표면
  - UI, 디버그, 검증 제약
- 다루지 않는 것:
  - 특정 종의 파라미터 값
  - 특정 종의 예외 상태명
  - 특정 종 전용 atlas stage 이름
  - 특정 종만 쓰는 보정 공식

## 입력 자료를 해석하는 방식

- 입력은 논문 요약, 행동 메모, 수치 표, 그래프 추정치, 서술형 설명일 수 있다.
- 어떤 형태의 입력이 오더라도 최종 출력은 반드시 구현 가능한 규칙과 파라미터여야 한다.
- 단순 요약으로 끝내지 말고, 최소한 아래 구조로 변환한다.
  - 개체가 무엇을 감지하는가
  - 어떤 조건에서 규칙이 발동하는가
  - 방향, 속도, 상태, 간격, 깊이, 회전이 어떻게 바뀌는가
  - 필요한 값, 범위, 단위가 무엇인가
  - 값이 부족하면 어떤 시작값과 튜닝 범위를 둘 것인가

## 최상위 행동 원칙

### 1. 개체 기반 규칙

- 모든 동물은 개체 기반으로 구현한다.
- 집단 행동은 각 개체가 주변 이웃과 환경에 반응한 결과로 나타나야 한다.
- 전역 중심점, 전역 형태, 전역 목표 벡터로 군집을 직접 조형하지 않는다.

### 2. 로컬 상호작용 우선

- 규칙은 가능한 한 로컬 상호작용으로 구성한다.
- 각 개체는 제한된 이웃, 제한된 지각 범위, 환경 입력만 본다고 가정한다.
- 위상적 상호작용이 핵심이면 neighbor count 축으로 해석한다.
- 거리 기반 상호작용이 핵심이면 perception radius, minimum spacing, preferred spacing 축으로 해석한다.

### 3. 단순 boid 3요소로 축약 금지

- alignment, cohesion, separation만으로 설명을 끝내지 않는다.
- 아래 항목은 필요 시 별도 규칙 축으로 분리한다.
  - 속도 조절
  - 회전 한계 또는 max turn rate
  - 반응 지연
  - 상태 전이
  - 포식자 반응
  - 회복 또는 재정렬
  - 환경 변화에 따른 규칙 변화
  - 개체차
  - 신호 전달 또는 참조 이웃 효과

### 4. 전역 연출보다 emergent behavior

- 화면상 보기 좋은 모양을 위해 전역 압축, 전역 방향 고정, 전역 집합 규칙을 추가하지 않는다.
- 로컬 규칙의 누적으로 밀도, 방향성, 분산, 재결합이 나오게 만든다.

## 수치 해석 원칙

### 5. 수치 우선

- 서술보다 수치를 우선한다.
- 허용되는 값의 형태:
  - 직접 수치
  - 범위
  - 그래프 추정치
  - 튜닝 시작값
- 빠름, 가까움, 강함, 자주, 조밀함 같은 표현은 조건과 수치로 바꾼다.

### 6. 값이 부족할 때

- 직접 수치가 있으면 그대로 사용한다.
- 그래프만 있으면 범위로 추정한다.
- 정성 묘사만 있으면 구현 규칙으로 번역한다.
- 그래도 부족하면 안전한 시작값과 튜닝 범위를 함께 둔다.
- 부족하다고 멈추지 말고, 근거 수준을 표시한 뒤 시뮬레이션 시작 후보를 제안한다.

### 7. 규칙 출력 형식

- 각 규칙은 최소한 아래 구조를 만족해야 한다.
  - 입력
  - 조건
  - 출력
  - 수치 또는 범위
  - 근거 수준
  - 구현 우선순위

## 상태와 환경 원칙

### 8. 상태 전이 필수

- 한 동물을 단일 평균 행동으로 취급하지 않는다.
- 가능한 상태 축:
  - 평상시
  - 위협 시
  - 먹이 탐색 시
  - 휴식 시
  - 낮
  - 밤
  - 번식기
  - 이동기
- 상태 전이는 다음을 포함해야 한다.
  - 발동 조건
  - 지속 시간
  - 해제 조건
  - 상태별 규칙 강도 변화

### 9. 환경 입력은 독립 변수

- 낮/밤, 수심, 고도, 조류, 빛, 장애물, 밀도, 지형은 배경 설명이 아니라 규칙 입력값이다.
- 환경은 서술이 아니라 modifier 또는 condition으로 연결한다.

### 10. 포식자 반응은 별도 시스템

- 포식자 반응을 separation 강화 한 줄로 축약하지 않는다.
- 아래를 분리해서 구현한다.
  - 감지 방식
  - 감지 거리
  - 즉각 회피 방향
  - 속도 증가 또는 감소
  - 회전 변화
  - 신호 전파 여부
  - 회피 후 recovery
  - recovery 후 재정렬 또는 재결합

### 11. 국소적이고 연속적인 회피

- 가능하면 포식자 반응은 국소적이고 연속적인 반응으로 해석한다.
- 화면 연출을 위해 무리 전체가 동시에 전역 반응하도록 만들지 않는다.

### 12. 개체차 허용

- 모든 개체가 같은 속도, 같은 임계값, 같은 민감도로 움직인다고 가정하지 않는다.
- 개체차는 필요한 축만 선택해서 넣는다.
  - 기본 속도
  - 반응 지연
  - 선호 간격
  - 회피 민감도
  - 노이즈 강도
  - 상태 전이 임계값

### 13. 공간 모델은 메커니즘 기준으로 정한다

- 화면이 2D여도 내부 행동 모델은 2D 또는 3D 중 메커니즘에 맞게 선택한다.
- 수직 분포나 depth 규칙이 핵심이면 내부 계산을 3D로 유지한다.
- 수평 평면 규칙이 핵심이면 2D 모델을 사용해도 된다.

### 14. 밀도와 간격은 별도 축

- 군집 밀도는 결과이자 동시에 규칙 입력이다.
- 최소한 아래 항목은 분리해서 판단한다.
  - minimum spacing
  - preferred spacing
  - perception radius 또는 neighbor count
  - 고밀도에서의 간격 변화
  - 위협 시 간격 변화

### 15. 전달 메커니즘 분리

- 방향 변화 전파, 경보, 페로몬, 발광 동기화, 접촉 전달은 alignment 하나로 합치지 않는다.
- 전달 거리, 전달 속도, 지연, 감쇠를 가능한 한 따로 뽑는다.

## 공용 구현 표면

이 프로젝트에서 행동과 표현은 분리한다.

- 행동 계층의 책임:
  - 이웃 탐색
  - 상태 전이
  - 방향, 속도, 위치 계산
  - 환경 반응
  - 포식자 반응
- 표현 계층의 책임:
  - stage 선택
  - frame 선택
  - 회전
  - 좌우 반전
  - atlas 기반 포즈 표현

### 행동 코드 위치

- 개체 기반 Home 동작은 src/behaviors/individual 안에서 관리한다.
- 시뮬레이션 동작은 src/behaviors/swarm 안에서 관리한다.
- 특정 동물의 행동 규칙은 해당 파일 안에 두되, sprite 전용 판단은 가능한 한 공용 계층으로 올린다.

### 공용 sprite 계층

- sprite atlas 메타데이터는 src/data/spriteAtlases.js 에 둔다.
- atlas frame, grid, sequence 계산은 src/utils/spriteAtlas.js 를 사용한다.
- pose, orientation, flip, rotation 계산은 src/utils/spritePose.js 를 사용한다.
- Home DOM atlas 렌더링은 src/components/SpriteAtlas.jsx 를 사용한다.

### 공용 sprite 사용 규칙

- Home DOM 경로는 resolveDomAtlasSprite 를 우선 사용한다.
- canvas 기반 placeholder 또는 2D canvas 렌더러는 resolveCanvasAtlasSprite 를 우선 사용한다.
- WebGL, p5 같은 커스텀 렌더러는 필요하면 resolveAtlasSpritePose 를 직접 써도 되지만, stage와 pose 판단은 같은 공용 atlas/pose 정의를 공유해야 한다.
- caller가 sprite stage, flip, rotation을 따로 흩어서 결정하지 않는다.
- Home과 Sim에 차이가 필요해도 그 차이는 atlas.pose.state, atlas.pose.resolve, profile 같은 중앙 정의에서 선언한다.
- 특정 종의 sprite-only 예외를 behavior 루프 여기저기에 흩뿌리지 않는다.

### placeholder와 이후 확장 규칙

- placeholder swarm은 지금 당장 단순한 움직임만 하더라도, 나중에 생성 로직이나 상태 로직이 붙을 수 있게 공용 sprite 경로를 유지한다.
- 새 생성 로직을 붙일 때는 필요하면 에이전트 상태에 아래 필드를 유지한다.
  - spriteProfile
  - spriteSpace
  - spriteState
  - previousScreenPosition
- 새 규칙이 생겨도 draw 코드에서 개별 stage 순회나 개별 flip 공식을 다시 만들지 않는다.

## 스프라이트와 시각 표현 규칙

### 16. 스프라이트는 표현 계층

- 스프라이트는 행동을 결정하는 계층이 아니라 계산된 상태를 표현하는 계층이다.
- 스프라이트 연출 때문에 행동 로직을 바꾸지 않는다.

### 17. 상태 기반 포즈 매핑

- 포즈와 프레임은 개체 상태에 연결한다.
  - 이동 방향
  - 회전량
  - 속도 변화
  - 수직 이동
  - 상태 전이
  - 보행, 유영, 비행 주기

### 18. 투영 후 방향 기준

- 좌우 포즈나 프레임 선택은 가능하면 실제 화면상 이동 방향 또는 투영 후 변화량을 기준으로 한다.
- raw 3D direction만으로 좌우 포즈를 고정하지 않는다.

### 19. 표현 일관성

- 좌우 반전, 회전, 프레임 선택 순서는 일관되어야 한다.
- 같은 종이 Home과 Sim에서 다른 sprite 해석 경로를 타지 않게 한다.
- 차이가 필요하면 caller 분기가 아니라 공용 pose 정의에서 선언한다.

### 20. 시각 효과로 행동 대체 금지

- 군집이 조밀해 보이게 하려고 스프라이트 크기, 알파, 레이어만 조정해서 행동 문제를 덮지 않는다.
- 행동은 행동 규칙으로 해결하고, 스프라이트는 그 결과를 따른다.

## 새 동물을 구현할 때의 최소 절차

1. 지배 메커니즘을 먼저 식별한다.
2. 입력 자료를 개체 규칙과 수치 파라미터로 번역한다.
3. 상태 축과 환경 modifier를 분리한다.
4. 포식자 반응이나 전달 메커니즘이 있으면 별도 시스템으로 뽑는다.
5. 내부 계산이 2D인지 3D인지 메커니즘 기준으로 정한다.
6. sprite는 행동 계산 뒤에 상태 매핑으로 연결한다.
7. 기존 패널 구조 안에서 필요한 최소 control만 남긴다.
8. 새 UI를 만들지 않는다.

## 최소 산출물 형식

새 동물이나 새 행동을 구현하기 전에 최소한 아래 정보를 확보한다.

- 핵심 메커니즘 3~5개
- 필수 규칙 5~8개
- 규칙 표 또는 규칙 목록
- 상태 전이 표
- 환경 modifier 목록
- 포식자 반응 표 또는 별도 규칙 목록
- 스프라이트 상태 매핑 규칙
- 불확실한 값과 튜닝 시작 범위

## 선택형 파라미터 축

아래 항목은 모든 동물에 강제로 적용하는 기본값 목록이 아니다.

- 파라미터는 전부 채우는 체크리스트가 아니라 선택형 메뉴로 다룬다.
- 해당 동물의 지배 메커니즘, 상태 구조, 환경 입력을 설명하는 데 필요한 축만 고른다.
- 쓰지 않는 축은 빈 값으로 남기지 말고 문서와 구현에서 아예 제외한다.
- 같은 이름의 축이라도 실제로 규칙에 연결되지 않으면 추가하지 않는다.
- 목표는 많은 파라미터를 나열하는 것이 아니라, 적은 수의 유효한 파라미터로 행동을 설명하는 것이다.

### 이동과 운동학 축

- body length 또는 characteristic size
- mean speed
- speed range
- burst speed
- acceleration
- turning radius 또는 max turn rate
- reaction delay
- noise amplitude

### 이웃 상호작용 축

- perception radius 또는 neighbor count
- minimum spacing
- preferred spacing
- alignment, cohesion, separation 계열 가중치
- density range

### 상태 축

- state types
- state transition condition
- state duration

### 포식자 반응 축

- predator detection distance
- predator impact distance
- escape speed multiplier
- escape turn multiplier
- regrouping 또는 recovery delay

### 공간과 환경 축

- vertical distribution 또는 altitude/depth range
- environmental modifiers

### 개체차 축

- individual variation range

### 선택 규칙

- 포식자 시스템이 없으면 predator 계열 파라미터는 넣지 않는다.
- 수직 분포가 중요하지 않으면 vertical distribution 계열 파라미터는 넣지 않는다.
- 위상적 상호작용이 핵심이면 perception radius보다 neighbor count를 우선 고려한다.
- 거리 기반 상호작용이 핵심이면 neighbor count보다 spacing, radius 계열을 우선 고려한다.
- 상태 전이가 단순하면 상태 파라미터도 최소 세트만 둔다.
- 환경이 행동을 거의 바꾸지 않으면 environmental modifiers를 억지로 만들지 않는다.

## UI 및 디버그 제약

### 21. 새 UI 생성 금지

- 명시적으로 요청받지 않은 한 새 패널, 새 오버레이, 새 디버그 창, 새 HUD, 새 그래프, 새 인포 박스를 만들지 않는다.
- 현재 시뮬레이션 호스트와 기존 컨트롤 패널 구조를 유지한다.

### 22. 최소 제어값만 노출

- 새 행동을 위해 필요한 값이 있더라도 기존 패널 구조 안에서 최소한만 노출한다.
- 구현에 꼭 필요하지 않으면 새 컨트롤을 추가하지 않는다.

### 23. 검증 수단 우선순위

- 새 UI보다 먼저 아래 순서로 검증한다.
  - 기존 파라미터 변화로 확인 가능한지
  - 코드 레벨 계산 경로로 확인 가능한지
  - 좁은 범위 오류 검사나 빌드로 확인 가능한지
- 임시 디버그 수단이 필요해도 상시 UI로 남기지 않는다.
- 구조 변경 후에는 가능한 한 npm run build 로 확인한다.

## Copilot 작업 순서

외부 행동 요약이나 논문 정리가 채팅에 들어오면 아래 순서로 처리한다.

1. 지배 메커니즘을 먼저 식별한다.
2. 자료를 개체 규칙, 상태, 환경 modifier, 수치 축으로 재구성한다.
3. 행동 계산과 sprite 표현을 분리한다.
4. 기존 공용 구현 표면을 우선 재사용한다.
5. 종별 sprite-only 분기가 필요하면 atlas.pose.state, atlas.pose.resolve, profile 쪽으로 모은다.
6. control은 최소한만 남긴다.
7. 새 UI는 만들지 않는다.
8. 좁은 검증 후 빌드로 확인한다.

## 금지 사항 요약

- 군집 중심점 기반 전역 제어
- 논문 근거 없는 연출용 값 남발
- 모든 동물을 동일한 보이드 파라미터 차이로만 해석하는 것
- 상태 전이 없는 평균 행동만 구현하는 것
- 수치 없이 행동 이름만 나열하는 것
- 스프라이트로 행동 문제를 가리는 것
- caller별 sprite 규칙 복제
- 명시적 요청 없는 새 UI 생성

## 유지보수 규칙

- 이 파일에는 완전 공통 규칙만 남긴다.
- 특정 동물에만 적용되는 내용은 이 파일에 추가하지 않는다.
- 종별 규칙이 필요하면 해당 동물 구현 파일, atlas 정의, 또는 별도 작업 문서에 둔다.
- 이 파일을 수정할 때는 공통 계약이 넓어졌는지, 아니면 특정 종의 예외가 섞인 것인지 먼저 구분한다.
